## 新架构

这部分内容在 `canrd-erp` 的 `docs/ARCHITECTURE_GUIDE.md` 里描述的很详细了，这里我只说说 ……

- 这套架构其实不是纯血缘正统的 DDD
  - 纯血的 DDD 是有 application 层，其作用是用于编排 domain 和 repository 从而实现完整的业务需求，但是由于本套系统要使用工作流来编排业务，所以其实也就替代了 application 层的功能，所以我在设计架构时就把 application 层砍掉了
  - 纯血的 DDD 中 domain 领域之间是不能直接调用的，是要通过领域事件进行通信的。但根据我所学知识来看，DDD 这么处理是为了微服务做准备，为了将来划分微服务更加便利，我认为我们公司目前的团队规模不太可能去往微服务考虑，而且加上领域事件要引入 redis 或者 kafka 等中间件，增加了系统复杂度，所以目前 domain 之间还是耦合的
